# MIDI Composition 極意

mido を使った SMF 生成・アレンジの実戦知識。**すべての失敗モードは実際のセッションで踏んだもの**なので、該当する検証を省略しないこと。

```bash
pip install mido --break-system-packages
```

## 鉄則(この順で守る)

1. **書く前に読む** — 既存曲を触るなら、まず構造を完全に把握してから1音でも書く
2. **1トラック = 1チャンネル** — 混在は多くのDAWでインポートエラーになる
3. **理論値ではなく実データに合わせる** — コードのルートは「はず」ではなく、実際に鳴っているベーストラックから読む
4. **生成したら必ず自動検証** — 耳で聴く前にスクリプトで異常を検出する

## 1. 解析フェーズ(アレンジ・修正の前に必須)

### 1-1. 構造ダンプ

最初に必ず全トラックの概要を出す:

```python
import mido
m = mido.MidiFile(path)
for i, t in enumerate(m.tracks):
    name = next((x.name for x in t if x.type=='track_name'), '')
    prog = {x.program for x in t if x.type=='program_change'}
    ch   = {x.channel for x in t if hasattr(x,'channel')}
    n    = sum(1 for x in t if x.type=='note_on' and x.velocity>0)
    print(i, name, 'prog', prog, 'ch', ch, 'notes', n)
```

**罠: トラック番号 ≠ チャンネル番号。** 「トラック5だからch5」と思い込むと、コピーしたノートと後付けCCのチャンネルが食い違い、1トラック内にch混在が発生する(実際に起きた)。チャンネルは必ずメッセージから読み取る。

### 1-2. テンポマップとグリッド

- トラック0の `set_tempo` を全部リストアップする。**曲頭に短い別テンポ区間(ピックアップ)があることがある**
- 小節グリッドの基準を確認: ベース等のノート開始tickが `4*tpb` の倍数に乗っているか。乗っていれば小節=1920tick(tpb=480時)の単純計算でよい

### 1-3. キーとコード進行の抽出

```python
# ピッチクラスのヒストグラム(ドラムch9除外、音価で重み付け)でキー推定
# 小節ごとのベース最低音 → ルートのピッチクラス列
```

- ヒストグラム上位からスケールを判定(例: B♭,E♭,C,A♭,F,G,D が強い → Cマイナー/E♭メジャー)
- **ルート列は必ず全小節ぶん出力して目視する。** ループ曲でも途中で転回・オクターブ下降・代理コード(G on Cm等)が挟まる

### 1-4. セクション検出

トラック×4小節ブロックのアクティビティマップ(`#`/`.`/空白)を出すと、イントロ/A/B/サビ/ブレイクが一目で分かる:

```
 2 |      ....####      ....####   |
11 |   .###############  ##########|
```

**ドラムやベースの「入り」の小節番号を必ず記録する。** アレンジ側で勝手に「12小節目から」等と決めると原曲とズレる(実際に9小節ズレた)。

## 2. 生成フェーズ

### 2-1. 絶対tick で組み、最後にデルタ変換

生成中はすべて `(絶対tick, Message)` のリストで持ち、最後にソートしてデルタ化する。デルタで直接組むとイベント挿入のたびに壊れる。

```python
events.sort(key=lambda x: (x[0], 0 if x[1].type=='note_off' else 1))  # 同tickはoffを先に
last = 0
for tk, msg in events:
    track.append(msg.copy(time=tk-last)); last = tk
```

**同tickの並び順**: note_off → control_change → program_change → note_on。offをonより先にしないと同音連打で音が消える。

### 2-2. チャンネル設計

先に**チャンネル台帳**を作ってから書く。ch9はドラム専用。既存トラックを残す場合はそのchを台帳に登録し、新規トラックは空きchのみ使う。

```
ch0 Violin / ch1 Hook / ch2 Bass(新) / ch3 Supersaw(新) / ... / ch9 Drums(新)
```

### 2-3. コード生成の落とし穴

- **ピッチクラス演算のバグに注意。** add9 を `62+(root+2)%12` のような式で書くと、mod が音名でなく鍵盤位置にかかってキー外の音が出る(Cm上でE♮が112回鳴った実例)。**テンションは「ルートのpc + 度数の半音数」を先に mod 12 してからオクターブを足す**:
  ```python
  ninth_pc = (root_pc + 2) % 12
  ninth = 60 + ninth_pc + (12 if ninth_pc < lowest_pc else 0)
  ```
- **マイナーキーのドミナント(G on Cm)のB♮は正しい音。** キー外検出で引っかかっても消さない
- **ボイスリーディング**: 転回形候補を列挙し、前の和音との音程移動合計が最小のものを選ぶ。毎回同じ積みだと手が飛んで機械臭くなる

### 2-4. リズム隊の実戦パターン(J-POP/EDM系)

| セクション | キック | スネア/クラップ | ハット |
|---|---|---|---|
| イントロ | 四つ打ちのみ | なし | 途中から8分・小さめ |
| Aメロ | 1拍・2.5拍 | クラップ2・4拍 | 8分 vel46 |
| Bメロ | 1・1.5・2.5拍 | クラップ2・4拍 | 8分・表拍アクセント |
| サビ | 四つ打ち vel118 | クラップ2・4拍 | 16分 + 裏拍オープンハット |
| ブレイク | 1拍のみ | 3拍(ハーフタイム) | 4分 |

- **フィルは「重ねる」のではなく「置き換える」。** フィル小節は3〜4拍目の通常パターン(キック・クラップ・ハット)を削ってからタムを置く。重ねるとゴチャつく(実際に指摘された)
- フィルの密度は行き先で変える: 盛り上がるセクションへ→16分8発タム回し、静かなセクションへ→8分4発
- クラッシュはセクション頭と8小節目

### 2-5. ベース

- Aメロ: ルート基準の跳ねグルーヴ(1, 2.5, 3, 4.5拍にオクターブ)
- B/サビ: 8分刻み + 小節末オクターブポップ、表拍 vel112 / 裏拍 vel96
- 音域は C1〜B2 に収める。`36 + root_pc` で47超えたら-12
- ゲートはステップ長-20tick程度空けて音を切る(ベタ伸ばしはモタる)

## 3. CCミックスパス

SMFで使える主要CC: **10=パン, 91=リバーブ, 93=コーラス, 11=エクスプレッション, 74=カットオフ(GS/XG/ソフトシンセ), 7=ボリューム**

### 定石

- **パン対称配置**: リード系はセンター、対になる楽器を左右対称に(例: ヴァイオリン44 ⟷ ピアノ84)。ベース・キックは必ずセンター
- **リバーブ**: 笛・ストリングス・FXは深め(90-105)、ベースはドライ(〜10)、ドラム軽め(35)
- **サイドチェイン風ポンピング**: サビのパッド/スーパーソーに CC11 を毎拍 `38→70→100→118→127` のカーブで書く。**セクションを抜けるとき CC11=127 にリセットするのを忘れない**(忘れると次の登場時に音が小さいまま)
- **オートパン**: アープに sin 波で CC10 を ±34、2小節周期
- **ライザー**: サビ直前8小節で CC74 を 40→127 に直線スイープ、サビ頭で64にリセット
- CC74 は素のGM音源では無視される。再生環境を確認してから使う

### 既存トラックへのCC追記

トラックを絶対tickに展開 → CCイベントをマージ → 再ソート → デルタ再構築。`end_of_track` を除外してから足し、最後に付け直す。

## 4. 検証フェーズ(保存後に必ず実行)

```python
m = mido.MidiFile(out_path)
scale = {0,2,3,5,7,8,10}  # 曲のキーに合わせる
for i, tr in enumerate(m.tracks):
    # (a) チャンネル混在
    chs = {x.channel for x in tr if hasattr(x,'channel')}
    assert len(chs) <= 1, f"track {i} mixed channels {chs}"
    # (b) キー外ノート(ch9除外)
    # → 検出されたら即修正ではなく「正当な音か」判定する
    #    正当: ドミナントの導音、原曲由来のクロマチックFX、経過音
    # (c) グリッド: 新規生成ノートは 16分 (tpb//4) の倍数tickに乗っているか
    # (d) 負のデルタタイムがないか
```

さらに:
- **チャンネルの重複**(別トラックが同chを使用)もチェック。プログラムチェンジが後勝ちで上書きされる
- 総尺 `m.length` が意図通りか
- ノートon/off対応(鳴りっぱなし検出): onスタックが最後に空か

## 5. 修正依頼への対応

- 「ズレてる」は**タイミング・キー・チャンネルのどれか分からない**。全部スキャンして原因を特定してから直す。ユーザーが名指ししたトラックが原因とは限らない(Piano Compのズレの正体がスーパーソーのadd9バグだった実例)
- ピンポイント修正は該当トラックだけ再構築し、他は触らない
- 修正のたびに §4 の検証を回す
